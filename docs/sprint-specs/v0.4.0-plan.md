## Sprint P5.1 — Dashboard Wire-Up + Embed + crates.io

**Requirements:** DE-01..DE-08  
**Tests:** T-DE-01..T-DE-06  
**Acceptance:** AC-11..AC-14

### Problem

`dashboard/index.html` is a placeholder — it does not load `team-dashboard.jsx`. The real dashboard UI exists (`dashboard/team-dashboard.jsx`, 41KB React component) but is never served. The daemon embeds the placeholder via `include_str!`, so users see a blank info page instead of the real dashboard.

Additionally, crates.io requires self-contained crates — no runtime file dependencies. The dashboard assets must be embedded in the binary at compile time **from within the crate root**.

### ⚠️ Blocking constraint (SCMUX-QA-P51-001)

**The current `include_str!("../../../dashboard/...")` path references files outside the crate root.** `cargo package` strips files not within `crates/scmux-daemon/`. This means DE-07/DE-08/AC-13 (crates.io publish) cannot be achieved without moving the assets.

**Required action:** Move dashboard assets inside the crate root before embedding:
```
crates/scmux-daemon/
  assets/
    index.html
    dashboard.js        ← compiled from team-dashboard.jsx
    react.min.js        ← production build (~43KB)
    react-dom.min.js    ← production build (~130KB)
```
The top-level `dashboard/` directory remains as the development source tree. During build, the compiled artifact is copied to `crates/scmux-daemon/assets/`.

### Visual Design Reference

The live look-and-feel is locked in at:

- **Reference JSX**: `dashboard/team-dashboard-original.jsx`
- **Working preview**: `http://localhost:8080/preview.html` (served from dashboard dir)
- **Design language**: dark background `#060810`, monospace (Berkeley Mono/Fira Code), per-project color accent bars, pulsing animated status dots, grid/list/grouped views, click-to-open jump modal with iTerm2/WezTerm button

The P5.1 implementation **must match this design exactly** when wiring live data.

### Approach

**Step 1 — Pre-compile JSX → `dashboard.js`**

JSX cannot run natively in browsers. Pre-compile using Node.js (proven working in preview.html):

```bash
# In dashboard/ directory:
node -e "
const babel = require('@babel/core');
const fs = require('fs');
const src = fs.readFileSync('team-dashboard.jsx', 'utf8')
  .replace(/^import\s+\{[^}]+\}\s+from\s+[\"']react[\"'];?\s*\n/m, '')
  .replace(/^export default function\s+/m, 'function ');
const result = babel.transformSync(src, {
  plugins: ['@babel/plugin-transform-react-jsx'],
  filename: 'dashboard.jsx'
});
fs.writeFileSync('dashboard.js',
  'const { useState } = React;\n\n' + result.code +
  '\n\nReactDOM.createRoot(document.getElementById(\"root\")).render(React.createElement(Dashboard));\n'
);
"
```

**Compile strategy (SCMUX-QA-P51-003 decision):** Commit `dashboard.js` as a pre-built artifact in `crates/scmux-daemon/assets/`. Add a CI check that reruns the compile and diffs to catch stale builds. `build.rs` approach deferred — it would require Node.js on all CI runners and complicates cross-compilation.

**React bundle strategy (SCMUX-QA-P51-006):** Use **production minified** builds:
- `react.production.min.js` (~43KB)
- `react-dom.production.min.js` (~130KB)

Total embedded payload: ~200KB vs 1.2MB for dev builds. Download production UMD builds from `unpkg.com/react@18/umd/react.production.min.js` and `unpkg.com/react-dom@18/umd/react-dom.production.min.js`.

**Step 2 — Replace index.html (SCMUX-QA-P51-005)**

The current `index.html` is a static info placeholder with no `<div id="root">`. Replace it:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <title>scmux</title>
  <script src="/react.min.js"></script>
  <script src="/react-dom.min.js"></script>
</head>
<body style="margin:0;background:#060810">
  <div id="root"></div>
  <script src="/dashboard.js"></script>
</body>
</html>
```

**Step 3 — Wire dashboard to live API (SCMUX-QA-P51-004)**

Replace the hardcoded `TEAMS` mock array with polling:
- `GET /sessions` → session list (fields are already compatible: `name`, `project`, `status`, `panes`, `session_ci`)
- `GET /hosts` → host reachability
- `GET /dashboard-config.json` → poll interval, default terminal
- Note: `pane.last_activity` is currently hardcoded to `"unknown"` in `tmux.rs`. Either fix `#{pane_activity}` collection or document as known gap for v0.4.0.

**Step 4 — Add static file routes + embed (SCMUX-QA-P51-002)**

The daemon currently only serves `GET /` with the embedded `index.html`. Add routes for all assets and implement `SCMUX_DASHBOARD_DIR` override:

```rust
// crates/scmux-daemon/src/api.rs

static INDEX_HTML: &[u8]    = include_bytes!("../assets/index.html");
static DASHBOARD_JS: &[u8]  = include_bytes!("../assets/dashboard.js");
static REACT_JS: &[u8]      = include_bytes!("../assets/react.min.js");
static REACT_DOM_JS: &[u8]  = include_bytes!("../assets/react-dom.min.js");

async fn serve_asset(path: &str, static_bytes: &[u8], content_type: &str) -> Response {
    if let Ok(dir) = std::env::var("SCMUX_DASHBOARD_DIR") {
        // Dev override: serve from disk
        match tokio::fs::read(format!("{dir}/{path}")).await {
            Ok(bytes) => ([(header::CONTENT_TYPE, content_type)], bytes).into_response(),
            Err(_) => StatusCode::NOT_FOUND.into_response(),
        }
    } else {
        ([(header::CONTENT_TYPE, content_type)], static_bytes).into_response()
    }
}

// Routes:
// GET /           → index.html
// GET /dashboard.js
// GET /react.min.js
// GET /react-dom.min.js
```

**Step 5 — crates.io publish (DE-07, DE-08)**

- Verify `cargo package` succeeds (assets inside crate root)
- Add `CRATES_IO_TOKEN` to repo secrets
- Add `cargo publish -p scmux && cargo publish -p scmux-daemon` steps to release workflow after binary builds

### Acceptance criteria (AC-11..AC-14)

- `http://localhost:7878/` shows the real scmux dashboard matching reference design (project colors, status dots, grid/list/grouped views, jump modal with terminal button)
- `cargo install scmux-daemon` on a clean machine serves the embedded dashboard
- Both `scmux` and `scmux-daemon` published to crates.io on each release tag
- `SCMUX_DASHBOARD_DIR=./dashboard cargo run` loads from disk for dev iteration

## Sprint P5.2 — Homebrew Auto-Update

**Requirements:** RA-01..RA-03  
**Acceptance:** AC-15

### Problem

Homebrew formula in `randlee/homebrew-tap` must be manually updated after each release. Should be automated.

### Approach

Add an `update-homebrew` job to `.github/workflows/release.yml` that runs after the `release` job (not `upload-release` — see SCMUX-QA-P52-002):

```yaml
update-homebrew:
  needs: [release]          # ← actual job name in release.yml
  runs-on: ubuntu-latest
  steps:
    - name: Download checksums
      env:
        GH_TOKEN: ${{ github.token }}
      run: gh release download ${{ github.ref_name }} -p checksums.txt

    - name: Parse SHAs and version
      run: |
        MACOS_SHA=$(grep aarch64-apple-darwin checksums.txt | awk '{print $1}')
        LINUX_SHA=$(grep x86_64-unknown-linux-gnu checksums.txt | awk '{print $1}')
        VERSION=${GITHUB_REF_NAME#v}
        echo "VERSION=$VERSION" >> $GITHUB_ENV
        echo "MACOS_SHA=$MACOS_SHA" >> $GITHUB_ENV
        echo "LINUX_SHA=$LINUX_SHA" >> $GITHUB_ENV

    - name: Checkout homebrew-tap
      uses: actions/checkout@v4
      with:
        repository: randlee/homebrew-tap
        token: ${{ secrets.HOMEBREW_TAP_TOKEN }}
        path: homebrew-tap

    - name: Patch formula and push
      run: |
        cd homebrew-tap
        sed -i "s/version \".*\"/version \"$VERSION\"/" Formula/scmux.rb
        sed -i "s|releases/download/v[^/]*/scmux-v[^-]*-aarch64|releases/download/v$VERSION/scmux-v$VERSION-aarch64|g" Formula/scmux.rb
        sed -i "s|releases/download/v[^/]*/scmux-v[^-]*-x86_64|releases/download/v$VERSION/scmux-v$VERSION-x86_64|g" Formula/scmux.rb
        # Patch SHAs — find the sha256 line after each URL
        python3 - <<'PY'
        import re, os
        rb = open('Formula/scmux.rb').read()
        mac_sha = os.environ['MACOS_SHA']
        linux_sha = os.environ['LINUX_SHA']
        rb = re.sub(r'(aarch64-apple-darwin.*?\n\s+sha256 ")[^"]*"', rf'\g<1>{mac_sha}"', rb, flags=re.DOTALL)
        rb = re.sub(r'(x86_64-unknown-linux-gnu.*?\n\s+sha256 ")[^"]*"', rf'\g<1>{linux_sha}"', rb, flags=re.DOTALL)
        open('Formula/scmux.rb', 'w').write(rb)
        PY
        git config user.email "github-actions[bot]@users.noreply.github.com"
        git config user.name "github-actions[bot]"
        git add Formula/scmux.rb
        git commit -m "scmux $VERSION"
        git push
```

**Prerequisites (one-time setup):**
- Create fine-grained PAT with write access to `randlee/homebrew-tap`, store as repo secret `HOMEBREW_TAP_TOKEN`
- Document in `PUBLISHING.md` secrets table alongside `CARGO_REGISTRY_TOKEN`

> Note: The Homebrew formula only covers `aarch64-apple-darwin` and `x86_64-unknown-linux-gnu`. `x86_64-apple-darwin` (Intel Mac) is not in the release matrix and not included (SCMUX-QA-P52-003). PUBLISHING.md should be updated to reflect this.

### Acceptance criteria (AC-15)

- Pushing a new version tag auto-updates the Homebrew formula within 5 minutes
- No manual intervention required after tagging

## Sprint P5.3 — ATM Integration (agent activity monitoring)

**Requirements:** ATM-01..ATM-07  
**Tests:** T-ATM-01..T-ATM-04  
**Acceptance:** AC-16

> ⚠️ **BLOCKED pending arch-cmux input** (SCMUX-QA-P53-001, SCMUX-QA-P53-003)
> 
> P5.3 cannot start until the following are resolved:
> 1. ATM daemon API endpoint + response schema
> 2. Session name → ATM agent matching rule definition
> 3. ATM daemon port convention (fixed or scmux.toml config field)

### Problem

scmux shows session exists and CI status, but not whether the Claude agent inside is active, idle, or stuck. ATM (agent-team-mail) daemon already tracks this for enrolled sessions.

### Open design questions (must resolve before sprint kickoff)

| # | Question | Needed for |
|---|----------|-----------|
| Q1 | ATM daemon API endpoint path and port? (e.g., `GET http://localhost:7700/agents`) | ATM-01, ATM-02, session_atm schema |
| Q2 | ATM agent state response schema? Fields: `session_name`/`agent_id`/`team`/`state`/`last_active_at`/`last_message_at` — do these match what ATM returns? | ATM-02, ATM-03 |
| Q3 | Session name → ATM agent matching rule? Options: (A) explicit `atm_agent` field in sessions table/config, (B) scmux.toml enrollment mapping, (C) daemon queries ATM for all agents and matches by session name pattern | ATM-01 |
| Q4 | ATM "stuck" detection — does ATM daemon report it directly, or is it a time threshold scmux applies? If time-based, what threshold? | ATM-04, dashboard indicator |
| Q5 | Is ATM port fixed or in scmux.toml? | config schema, ATM-01 |

### Approach (tentative, pending Q1–Q5)

1. Add `session_atm` table to db.rs migrate():
   ```sql
   CREATE TABLE IF NOT EXISTS session_atm (
     session_name TEXT NOT NULL,
     agent_id     TEXT,
     team         TEXT,
     state        TEXT NOT NULL DEFAULT 'unknown',
     last_active_at  TEXT,
     last_message_at TEXT,
     updated_at   TEXT NOT NULL
   );
   ```
   > Schema coupling: use session name (TEXT) not FK to sessions.id — intentional loose coupling for ATM enrollment model (SCMUX-QA-P53-002).

2. Add separate `tokio::spawn` poll loop for ATM (mirrors CI poll pattern, 15s interval). Keep separate from session poll to avoid NF-03 budget impact (SCMUX-QA-P53-004).

3. `GET /sessions` includes `atm: {state, last_active_at} | null` per session.

4. Dashboard activity badge using same dot/status design language as existing status dots.

5. `scmux list` adds activity column (only shown for ATM-enrolled sessions).

6. Non-ATM sessions: `atm: null`. If ATM daemon unreachable: `atm: null`, no crash, no error in response.

### Acceptance criteria (AC-16)

- ATM-enrolled sessions show agent state in dashboard and CLI
- Non-ATM sessions unaffected
- ATM daemon unavailable = graceful degradation (null atm field, no daemon crash)
