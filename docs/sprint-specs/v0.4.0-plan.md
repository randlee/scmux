## Sprint P5.1 — Dashboard Wire-Up + Embed + crates.io

**Requirements:** DE-01..DE-08  
**Tests:** T-DE-01..T-DE-06  
**Acceptance:** AC-11..AC-14

### Problem

`dashboard/index.html` is a placeholder — it does not load `team-dashboard.jsx`. The real dashboard UI exists (`dashboard/team-dashboard.jsx`, 41KB React component) but is never served. The daemon embeds the placeholder via `include_str!`, so users see a blank info page instead of the real dashboard.

Additionally, crates.io requires self-contained crates — no runtime file dependencies. The dashboard assets must be embedded in the binary at compile time.

### Visual Design Reference

The live look-and-feel is locked in at:

- **Reference JSX**: `dashboard/team-dashboard-original.jsx`
- **Working preview**: `http://localhost:8080/preview.html` (served from dashboard dir)
- **Design language**: dark background `#060810`, monospace (Berkeley Mono/Fira Code), per-project color accent bars, pulsing animated status dots, grid/list/grouped views, click-to-open jump modal with WezTerm button

The P5.1 implementation **must match this design exactly** when wiring live data.

### Approach

**Step 1 — Pre-compile JSX to plain JavaScript**

`team-dashboard.jsx` uses JSX syntax which browsers cannot run natively. The correct approach (proven working in preview.html) is to pre-compile at build time:

```bash
# In dashboard/ directory:
node -e "
const babel = require('@babel/core');
const fs = require('fs');
const src = fs.readFileSync('team-dashboard.jsx', 'utf8')
  .replace(/^import\s+\{[^}]+\}\s+from\s+[\"']react[\"'];?\s*\n/m, '')
  .replace(/^export default function\s+/m, 'function ');
const result = babel.transformSync(src, {
  plugins: ['@babel/plugin-transform-react-jsx'],
  filename: 'dashboard.jsx'
});
fs.writeFileSync('dashboard-compiled.js',
  'const { useState } = React;\n\n' + result.code +
  '\n\nReactDOM.createRoot(document.getElementById(\"root\")).render(React.createElement(Dashboard));\n'
);
"
```

- No CDN dependency, no Babel in browser, no importmap
- `@babel/core` and `@babel/plugin-transform-react-jsx` are dev-only Node.js dependencies
- Output is plain `React.createElement(...)` calls readable by any browser

**Step 2 — Update index.html to load compiled assets**

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8" />
  <title>scmux</title>
  <script src="/react.js"></script>
  <script src="/react-dom.js"></script>
</head>
<body>
  <div id="root"></div>
  <script src="/dashboard.js"></script>
</body>
</html>
```

- Daemon serves `react.js`, `react-dom.js`, and `dashboard.js` as static files
- React UMD builds are embedded alongside dashboard JS (no CDN)

**Step 3 — Wire dashboard to live API**

Replace the hardcoded `TEAMS` mock array in `team-dashboard.jsx` with polling:
- `GET /sessions` → session list with pane status, CI, ATM (P5.3)
- `GET /hosts` → host reachability
- `GET /dashboard-config.json` → poll interval, default terminal
- Poll interval from `dashboard-config.json` → `poll_interval_ms`
- Unreachable host sessions render in monochrome (existing requirement DH-01..DH-04)
- Session structure must map cleanly to the existing card/row/modal components

**Step 4 — Embed via `include_str!` / `rust-embed`**

```rust
// Option A: include_str! (simpler, fine for < 1MB)
static INDEX_HTML: &str = include_str!("../../dashboard/index.html");
static DASHBOARD_JS: &str = include_str!("../../dashboard/dashboard-compiled.js");
static REACT_JS: &str = include_str!("../../dashboard/react.js");
static REACT_DOM_JS: &str = include_str!("../../dashboard/react-dom.js");

// Option B: rust-embed (directory-based, easier to add files later)
#[derive(RustEmbed)]
#[folder = "../../dashboard/dist/"]
struct DashboardAssets;
```

- Add `SCMUX_DASHBOARD_DIR` env var: if set, serve from disk instead of embedded
- This enables `SCMUX_DASHBOARD_DIR=./dashboard cargo run` for local dev iteration

**Step 5 — crates.io publish**

- Verify `cargo package --no-verify` succeeds
- Add `CRATES_IO_TOKEN` to repo secrets
- Add `cargo publish -p scmux && cargo publish -p scmux-daemon` steps to release workflow after binary builds
- `scmux` depends on no embedded assets; `scmux-daemon` embeds dashboard at compile time

### Acceptance criteria (AC-11..AC-14)

- `http://localhost:7878/` shows the real scmux dashboard matching reference design
- `cargo install scmux-daemon` on a clean machine serves the embedded dashboard
- Both `scmux` and `scmux-daemon` published to crates.io on each release tag
- `SCMUX_DASHBOARD_DIR` override loads from disk for local dev

### Build prerequisites

The pre-compile step requires Node.js and two npm packages:
```bash
cd dashboard && npm install --save-dev @babel/core @babel/plugin-transform-react-jsx
node compile.js  # or inline script above
```

This step must run before `cargo build`. Add to `Makefile` or document in `PUBLISHING.md`.

### Open questions for design review

1. **React bundle size**: react.js (107KB) + react-dom.js (1MB) embedded in binary. Acceptable? Alternative: CDN with offline fallback?
2. **`rust-embed` vs `include_str!`**: rust-embed adds a dependency but makes directory management easier. Worth it for 4 files?
3. **Session data shape**: `team-dashboard.jsx` uses `{name, project, sessionStatus, prs, panes}`. Does `/sessions` already return compatible fields, or do we need a mapping layer?
4. **Compile step automation**: Should `dashboard-compiled.js` be committed to git (pre-built artifact) or generated at build time via a `build.rs` script?

## Sprint P5.2 — Homebrew Auto-Update

**Requirements:** RA-01..RA-03  
**Acceptance:** AC-15

### Problem

Homebrew formula in `randlee/homebrew-tap` must be manually updated after each release. Should be automated.

### Approach

Add a job to `.github/workflows/release.yml` that runs after all build/upload jobs:

```yaml
update-homebrew:
  needs: [build-macos, build-linux, upload-release]
  runs-on: ubuntu-latest
  steps:
    - name: Download checksums
      run: gh release download ${{ github.ref_name }} -p checksums.txt

    - name: Parse SHAs
      run: |
        MACOS_SHA=$(grep aarch64-apple-darwin checksums.txt | awk '{print $1}')
        LINUX_SHA=$(grep x86_64-unknown-linux-gnu checksums.txt | awk '{print $1}')
        VERSION=${GITHUB_REF_NAME#v}
        echo "VERSION=$VERSION" >> $GITHUB_ENV
        echo "MACOS_SHA=$MACOS_SHA" >> $GITHUB_ENV
        echo "LINUX_SHA=$LINUX_SHA" >> $GITHUB_ENV

    - name: Update homebrew-tap
      uses: actions/checkout@v4
      with:
        repository: randlee/homebrew-tap
        token: ${{ secrets.HOMEBREW_TAP_TOKEN }}
        path: homebrew-tap

    - name: Patch formula and push
      run: |
        cd homebrew-tap
        sed -i "s/version \".*\"/version \"$VERSION\"/" Formula/scmux.rb
        sed -i "s|releases/download/v[^/]*/scmux-v[^/]*-aarch64|releases/download/v$VERSION/scmux-v$VERSION-aarch64|" Formula/scmux.rb
        sed -i "s|releases/download/v[^/]*/scmux-v[^/]*-x86_64|releases/download/v$VERSION/scmux-v$VERSION-x86_64|" Formula/scmux.rb
        sed -i "/aarch64-apple-darwin/{ n; s/sha256 \".*\"/sha256 \"$MACOS_SHA\"/ }" Formula/scmux.rb
        sed -i "/x86_64-unknown-linux-gnu/{ n; s/sha256 \".*\"/sha256 \"$LINUX_SHA\"/ }" Formula/scmux.rb
        git config user.email "github-actions[bot]@users.noreply.github.com"
        git config user.name "github-actions[bot]"
        git add Formula/scmux.rb
        git commit -m "scmux $VERSION"
        git push
```

**Secrets required:** `HOMEBREW_TAP_TOKEN` (fine-grained PAT with `randlee/homebrew-tap` write access)

### Acceptance criteria (AC-15)

- Pushing a new version tag auto-updates the Homebrew formula within 5 minutes
- No manual intervention required after tagging

## Sprint P5.3 — ATM Integration (agent activity monitoring)

**Requirements:** ATM-01..ATM-07  
**Tests:** T-ATM-01..T-ATM-04  
**Acceptance:** AC-16

### Problem

scmux shows session exists and CI status, but not whether the Claude agent inside is active, idle, or stuck. ATM (agent-team-mail) daemon already tracks this for enrolled sessions.

### Approach

1. Daemon queries ATM daemon API (`http://localhost:<atm_port>/agents`) for agent state.
2. Stores in new `session_atm` table: `session_name, agent_id, team, state, last_active_at, last_message_at`.
3. `/sessions` API includes `atm` field (state object or null) per session.
4. Dashboard shows activity indicator (active/idle/stuck) for ATM sessions using same dot/badge design language.
5. CLI `scmux list` shows activity column.
6. Non-ATM sessions fall back to `pane_last_activity` from tmux.

**Open questions for design review:**
1. What is the ATM daemon API endpoint and response schema? Need to confirm with arch-cmux.
2. ATM port: is it fixed or in config? Should `scmux.toml` have an `atm_port` field?
3. "Stuck" detection: threshold for idle→stuck transition? Configurable?
4. Should ATM poll happen in the same task loop as session poll, or a separate task?

### Acceptance criteria (AC-16)

- ATM-enrolled sessions show agent state in dashboard and CLI
- Non-ATM sessions unaffected
- ATM daemon unavailable = graceful degradation (null atm field, no crash)
